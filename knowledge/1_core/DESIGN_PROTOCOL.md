# 🎨 DESIGN PROTOCOL (V1.0)

Governs the visual and aesthetic "Soul" of all Antigravity projects using the Google `DESIGN.md` specification.

## ⚖️ 1. THE DUAL-LAYER MANDATE
Every project MUST utilize a two-pronged governance architecture:
1.  **BLUEPRINT.md**: Authoritative for **Function & Structure**.
2.  **DESIGN.md**: Authoritative for **Visuals & Aesthetics**.

## 🧬 2. DESIGN.MD STRUCTURE
All `DESIGN.md` files MUST follow the standard Google Agentic Design format:

### 2.1 YAML Tokens (The DNA)
- **Primary Colors**: Authority for brand identity.
- **Typography**: Definition of scale, weight, and font-family.
- **Spacing/Radius**: Definition of the grid and component curvature.
- **Atmospherics**: (e.g., Glassmorphism, Blur levels, Gradients).

### 2.2 Markdown Principles (The Spirit)
- **Visual Identity**: Narrative description of the "feeling" (e.g., Cyber Luxury).
- **Anti-Slop Standards**: Specific rules to prevent generic AI UI (e.g., "No default blue buttons," "Use only custom HSL gradients").
- **Component Guidelines**: Specific instructions for implementing complex UI elements.

## 🛠️ 3. OPERATIONAL RULES
1.  **Token-First Engineering**: AI MUST read the `design_tokens` block before writing any CSS or UI code.
2.  **The Design-Verifier Loop**: UI code generated by the AI must be audited against the `DESIGN.md` principles before finalization.
3.  **Baton Relay Sync**: Any change to the visual identity in a chat session MUST be synced back to the `DESIGN.md` Change Log.

## 🔗 4. BLUEPRINT SYNERGY (THE FUSION RULE)
1.  **Feature-to-Visual Mapping**: For every high-priority feature defined in `BLUEPRINT.md`, a corresponding `design_component` MUST be defined in `DESIGN.md`.
2.  **Constraint Sync**: Visual constraints (e.g., accessibility needs) in `BLUEPRINT.md` MUST dictate token shifts in `DESIGN.md`.
3.  **Dynamic Expansion**: When AI develops a superior UI pattern during a task, it MUST propose an "Aesthetic Patch" to the project's `DESIGN.md` to persist the learning.
